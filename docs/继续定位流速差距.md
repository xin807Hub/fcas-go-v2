## 新结论

这次剩下的差距，已经不是“总量重复记账”了，而是 **平均流速的定义不一致**：

- `v1` 的平均流速口径是：`总流量 * 8 / 整段查询时长`
- `v2` 之前的平均流速口径是：`avg(每个有数据时间片的 traffic_*_bps)`

这两个口径只有在“每个时间片都有数据”时才会接近；一旦有不少时间片没有该运营商的数据，`v2` 就会因为 **分母变小** 而明显偏大。

## 代码依据

### v1 的平均流速怎么算

`v1` 不是直接对时间片流速做 `avg()`，而是在 Java VO 里动态计算：

- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\java\com\marfosec\modules\traffic\entity\TrafficRankTrendVoEntity.java`

核心逻辑：

- `avgAppTrafficDnSpeed = appTrafficDn * 8 / timeSpan`
- `avgAppTrafficUpSpeed = appTrafficUp * 8 / timeSpan`

其中 `timeSpan` 不是“有数据的时间片数”，而是：

- 按查询开始、结束时间先算整段时长
- 再按当前颗粒度向上取整
- 10 分钟粒度就按 `600s` 对齐
- 1 小时粒度就按 `3600s` 对齐

也就是说，`v1` 的“平均”本质上是：

**查询窗口内的整段平均吞吐**

### v2 之前为什么会偏大

`v2` 一级汇总和一级表格之前都在做：

- `avg(traffic_up_bps)`
- `avg(traffic_dn_bps)`

对应位置：

- `server/service/traffic/common_traffic.go`
- `server/service/traffic/isp_traffic.go`

这里的 `traffic_*_bps` 是“有结果行的时间片流速”。  
如果某个 ISP 在部分时间片没有数据，这些时间片根本不会出现在结果里，也就不会参与 `avg()`。

于是实际效果变成了：

**只对活跃时间片求平均**

这一定会大于或等于“整段查询时长平均”。

## 从常理判断，哪个更合理

如果页面文案写的是“查询时间范围内的上行平均 / 下行平均”，那更合理的口径应该是：

**总量 ÷ 整段查询时长**

因为用户看到的是一个完整时间范围，例如：

- `2026-04-15 00:00:00 - 2026-04-16 00:00:00`

这类表达在常规理解里，平均值就应该覆盖整整 `24h`，而不是“只统计这个运营商出现过数据的那些时间片”。

### 用你这组 24 小时样例直接反推

`v1`：

- 上行总量：`56.74 TB`
- 下行总量：`109.35 TB`
- 查询时长：`24h = 86400s`

按 `总量 * 8 / 86400` 粗算：

- 上行平均约 `5.25 Gbps`
- 下行平均约 `10.13 Gbps`

这和 `v1` 页面显示几乎完全一致。

再看 `v2`：

- 上行总量：`56.83 TB`
- 下行总量：`109.38 TB`

如果也按完整 `24h` 去算，结果仍然应该接近：

- 上行平均约 `5.26 Gbps`
- 下行平均约 `10.13 Gbps`

但 `v2` 显示成了：

- 上行平均：`7.36 Gbps`
- 下行平均：`14.16 Gbps`

这两个值反推回去，相当于它实际只用了更短的“有效时长”做分母：

- 上行大约按 `17.1h`
- 下行大约按 `16.9h`

这正是“只按活跃时间片平均”的典型特征。

所以从常理和数学上看，这组数据里：

**`v1` 的平均流速更合理，`v2` 的平均流速偏大。**

## 本次修正

这次已经把 `v2` 一级汇总/一级表格的平均流速口径改回和 `v1` 一致：

- `server/service/traffic/common_traffic.go`
- `server/service/traffic/isp_traffic.go`
- `server/service/traffic/app_type_traffic.go`
- `server/service/traffic/app_id_traffic.go`

修正后的口径是：

- 先汇总 `sum(traffic_up)` / `sum(traffic_dn)`
- 再按 `查询总时长(向颗粒度上取整)` 计算平均流速

公式：

- `avg_up_bps = sum(traffic_up) * 8 / totalSpanSeconds`
- `avg_dn_bps = sum(traffic_dn) * 8 / totalSpanSeconds`
