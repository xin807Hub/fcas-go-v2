# 定位流量差距结论

## 结论

基于 `docs/定位流量差距.md` 里的截图数据，以及 `v1 / v2` 两套代码的静态对比，可以先下一个明确结论：

`v2` 不是“显示格式不同”或“单位换算不同”，而是**底层查询口径被整体放大了**。这次样例里，`v2` 的上行平均、下行平均、上行总量、下行总量、上下行总量，全部都比 `v1` 高约 `49.7%`，且放大量级几乎完全一致。

这说明问题发生在**流量明细聚合阶段**，不是前端展示层，也不是某一个单独字段的格式化问题。

## 差异量化

查询条件来自原始文档：

- 查询时间：`2026-04-15 00:00:00 ~ 2026-04-15 10:58:00`
- 查询链路：`1` 条链路

截图中两版结果对比如下：

| 指标 | v1 | v2 | 差异 |
| --- | --- | --- | --- |
| 上行平均 | 6.11 Gbps | 9.14 Gbps | +49.59% |
| 下行平均 | 12.03 Gbps | 18.01 Gbps | +49.71% |
| 上行总量 | 30.23 TB | 45.25 TB | +49.69% |
| 下行总量 | 59.56 TB | 89.17 TB | +49.71% |
| 上下行总量 | 89.79 TB | 134.42 TB | +49.70% |

这个特征非常关键：**不是某一列偏差，而是整套结果都按接近 `1.5x` 的比例被放大**。

## 主因判断

### 1. `v1` 与 `v2` 的“用户侧字段”和“运营商字段”配对方向不一致

`v1` 的运营商查询逻辑在 XML 里是这样的：

- `user_id` 分支取的是 `d_isp as name`
- `d_user_id` 分支取的是 `isp as name`

对应位置：

- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\mapper\traffic\DwsIsp10minDao.xml:32`
- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\mapper\traffic\DwsIsp10minDao.xml:43`
- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\mapper\traffic\DwsIsp10minDao.xml:81`
- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\mapper\traffic\DwsIsp10minDao.xml:92`

也就是：

- 命中 `user_id` 时，统计对端运营商 `d_isp`
- 命中 `d_user_id` 时，统计对端运营商 `isp`

而 `v2` 当前实现正好相反：

- `user_id` 分支配的是 `isp`
- `d_user_id` 分支配的是 `d_isp`

对应位置：

- `server/service/traffic/isp_traffic.go:87`
- `server/service/traffic/isp_traffic.go:88`
- `server/service/traffic/isp_traffic.go:101`
- `server/service/traffic/isp_traffic.go:103`

这会带来两个直接后果：

1. 查询维度从 `v1` 的“对端运营商”偏成了“本端运营商”。
2. 同一批流量落入 `UNION DISTINCT` 两个分支时，更不容易被判定为同一条记录，从而更容易被重复累计。

从当前样例看，`v2` 全指标几乎统一放大到 `1.5x`，很符合“有一部分流量被重复入账”的特征。

### 2. `v1` 在 `UNION DISTINCT` 之后还有一次按时间片聚合，`v2` 没有补这一层

`v1` 在两个分支 `UNION DISTINCT` 之后，还有一层：

- `GROUP BY name,start_time,is_oversea`

对应位置：

- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\mapper\traffic\DwsIsp10minDao.xml:120`

`v2` 当前在两个分支 `Union Distinct` 之后，直接把结果向上返回：

- `server/service/traffic/isp_traffic.go:115`

后续再在表格/汇总层做 `sum(traffic_up)`、`sum(traffic_dn)`、`avg(traffic_up_bps)`。

这意味着：

- 同一个 `isp + start_time` 如果在两个分支都出现，但数值不完全相同，会以两行继续向上游传递；
- 总量会继续被累计；
- 平均值和峰值也会被拉偏。

这条更像是**放大偏差的次因**。真正让 `v2` 和 `v1` 口径不一致的核心，还是前一条“字段方向配错”。

## 前端不是主因，但汇总口径也有差异

`v1` 页面汇总是把表格行逐条相加后展示的：

- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\vue\src\views\modules\traffic\dwsispmodule2.vue:270`
- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\vue\src\views\modules\traffic\dwsispmodule2.vue:272`
- `D:\code\fcas-java-v1\webapp-fcas-latest\src\main\resources\vue\src\views\modules\traffic\dwsispmodule2.vue:277`

`v2` 页面汇总直接吃后端 `gatherData`：

- `frontend/src/view/flowMonitor/ispRank.vue:387`
- `frontend/src/view/flowMonitor/ispRank.vue:397`

这不是本次 `+49.7%` 的主因，因为总量本身在后端已经偏大了；但它会让后续排查时更难用“表格合计”和“汇总展示”做交叉验证。

## 建议修复

### 必修

把 `server/service/traffic/isp_traffic.go` 的查询方向改回和 `v1` 一致：

- `user_id` 分支应取 `d_isp`
- `d_user_id` 分支应取 `isp`

同时要一起检查并同步这些过滤条件的方向：

- `param.Isp`
- `param.IsOversea`
- 后续若恢复/补充 `ispNameList`，也要同样按这个方向处理

否则即使 select 字段改对，筛选口径仍然会和 `v1` 不一致。

### 建议一起改

在 `Union Distinct` 之后，补一层按 `isp,start_time` 的再次聚合，再进入上层汇总/表格计算。这样可以把同一时间片的同名运营商先合并掉，避免平均值、峰值继续被放大。

## 验收建议

修完后，直接用原文档这组条件回归：

- 时间：`2026-04-15 00:00:00 ~ 2026-04-15 10:58:00`
- 链路：`1` 条

验收标准建议设成：

- `v2` 的上/下行总量与 `v1` 偏差控制在 `1%` 以内
- `v2` 的上/下行平均与 `v1` 偏差控制在 `1%` 以内
- 若仍有偏差，再去查 `v2` 数据表本身是否已经在离线层做过不同的预聚合

## 备注

这份结论是基于现有截图数值和代码静态比对得到的，**没有直接对数据库执行现场 SQL**。但从代码差异和结果放大比例看，当前最可信的主因就是：

**`v2` 的运营商维度取值方向与 `v1` 不一致，导致部分流量更容易重复记账。**

## 2026-04-15 修复补充

### 1. 现场 SQL 复盘后，问题分成了两个层次

这次按同一组真实条件反复回放后，发现问题不只是“`v1 / v2` 字段方向不一致”，还叠加了 **`v2` 新表落库语义变化**：

- `v1` 的运营商一级排名长期使用固定双分支：
  - `user_id -> d_isp`
  - `d_user_id -> isp`
- `v2` 的 `dws_isp_10m` 每行已经同时带有 `user_id / d_user_id / isp / d_isp`
- 在 **没有 `userIdList` 约束**、只按 `link_id` 查全量时，如果仍像 `v1` 一样把两支都 `UNION DISTINCT` 出来，就会把同一批流量按两侧 ISP 各算一遍

这个现象在现场结果里非常明显：

- 双分支直接查：`289.36 TB`，接近正确值 `147.31 TB` 的 `2x`
- 只保留 `d_isp` 单分支：`72.34 TB`，接近正确值的一半

所以这次真正的根因不是“应该取左边还是右边”，而是：

**`v2` 在无用户视角时，不能再直接复用 `v1` 的双分支 UNION 口径；必须先按每条明细选出唯一的“对端 ISP”。**

### 2. 中间几次误判和对应现象

这次排查里一共踩到了 3 类典型问题：

1. 直接双分支恢复成 `v1` 结构后，结果接近 `2x`
   说明 `v2` 新表里同一批流量已经能同时命中 `isp / d_isp` 两侧，继续双分支会重复记账。

2. 为了避开双计，简单改成只查 `d_isp` 单分支后，结果接近 `1/2`
   说明正确口径也不是“永远只看一侧 ISP”。

3. 在后续修正过程中，还出现了两个 ClickHouse 语法/聚合问题：
   - `code: 184`
     原因是 post-union 聚合结果直接起名为 `traffic_up/traffic_dn`，上层又继续 `sum(traffic_up)`，被 ClickHouse 判定为“聚合套聚合”。
   - `code: 215`
     原因是直接在聚合层使用 `if(user_id != 0, d_isp, isp)` 做分组表达式，ClickHouse 仍把 `user_id` 视为未正确进入分组键。

### 3. 最终修复思路

最终方案分成两条口径：

- 有 `userIdList` 时：
  继续保留 `v1` 的双分支逻辑，因为这时查询目标就是“用户视角下的对端 ISP”，两支分别对应：
  - `user_id -> d_isp`
  - `d_user_id -> isp`

- 没有 `userIdList` 时：
  不再做双分支 `UNION DISTINCT`，而是对 `v2` 表里的每条明细先选出唯一 ISP：
  - `if(user_id != 0, d_isp, isp) AS selected_isp`
  - 再按 `selected_isp,start_time` 聚合

这样做的含义是：

- 如果当前行是“源用户侧”流量，就取对端运营商 `d_isp`
- 否则取 `isp`
- 同一条明细只会落入一个运营商，不会再同时进入两支

### 4. 代码层怎么改的

本次代码调整集中在 `server/service/traffic/isp_traffic.go`：

- `getIspDb`
  按“有无 `userIdList`”拆成两条查询路径。

- `buildLegacyIspDb`
  保留 `v1` 风格的双分支查询，并统一修正了方向：
  - `user_id` 分支取 `d_isp`
  - `d_user_id` 分支取 `isp`

- `buildV2IspDbWithoutUserScope`
  专门处理 `v2` 在无用户视角下的一级运营商查询：
  - 先用 `if(user_id != 0, d_isp, isp)` 生成 `selected_isp`
  - 再在外层按 `selected_isp,start_time` 聚合

- `buildIspTrafficQuery`
  把 post-union 聚合和最终字段展开抽成公共函数，并保留一层中间别名：
  - `grouped_traffic_up`
  - `grouped_traffic_dn`
  这样可以避开 ClickHouse 的聚合嵌套报错。

- `applyNonEmptyIspFilter / applyNonEmptyIspExprFilter`
  表格、汇总、无用户口径统一过滤空运营商，确保 `isp = ''` 不再进结果。

### 5. 为了绕过 ClickHouse 报错，额外做的两个处理

#### 5.1 规避 `code: 184`

聚合后不再直接输出：

- `sum(traffic_up) AS traffic_up`
- `sum(traffic_dn) AS traffic_dn`

而是先输出：

- `sum(traffic_up) AS grouped_traffic_up`
- `sum(traffic_dn) AS grouped_traffic_dn`

再包一层把字段展开回：

- `grouped_traffic_up AS traffic_up`
- `grouped_traffic_dn AS traffic_dn`

这样上层表格和汇总继续 `sum(traffic_up)` 时，不会被 ClickHouse 识别成“聚合函数套聚合函数”。

#### 5.2 规避 `code: 215`

`if(user_id != 0, d_isp, isp)` 不能直接在聚合层里又 `SELECT` 又 `GROUP BY`。  
最终做法是先物化一层子查询：

- 内层先产出 `selected_isp,start_time,bytes_up_view,bytes_dn_view`
- 外层再 `GROUP BY selected_isp,start_time`

对应 SQL 形态如下：

```sql
SELECT selected_isp AS isp,
       start_time,
       sumMerge(bytes_up_view) AS traffic_up,
       sumMerge(bytes_dn_view) AS traffic_dn
FROM (
  SELECT if(user_id != 0, d_isp, isp) AS selected_isp,
         start_time,
         bytes_up_view,
         bytes_dn_view
  FROM bigdata_fcas_v2.dws_isp_10m
  ...
)
GROUP BY selected_isp, start_time
```

### 6. 这次修完后的验收标准

一级运营商汇总在这组条件下应回到与 `v1` 一致的量级：

- 时间：`2026-04-15 00:00:00 ~ 2026-04-15 15:25:57`
- 链路：`link_id = 1`

目标值：

- 上行平均：`7.15 Gbps`
- 下行平均：`13.74 Gbps`
- 上行总量：`50.44 TB`
- 下行总量：`96.87 TB`
- 上下行总量：`147.31 TB`

验收时重点看两件事：

1. 无 `userIdList` 的 `v2` SQL 中，不应再出现 `Union Distinct`
2. SQL 中应出现：
   - `if(user_id != 0, d_isp, isp) AS selected_isp`
   - `SELECT selected_isp AS isp ... GROUP BY selected_isp,start_time`

### 7. 最终结论

这次问题的最终结论可以收敛成一句话：

**`v1` 的双分支查询逻辑本身没有错，但 `v2` 的 `dws_isp_10m` 已经不是完全相同的落库语义；在无用户视角下，必须先按行选出唯一对端 ISP，再聚合，否则就会出现整批流量接近 `2x` 的重复记账。**
